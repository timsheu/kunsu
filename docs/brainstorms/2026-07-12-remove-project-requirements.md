# 移除子專案登記：軍師端 remove-project 子指令

## 摘要

`kunsu-init` 新增 `remove-project` 子指令，於軍師 repo 內執行：列出本軍師目前登記的子專案（路徑已不存在的標記警示並排前面），選定後整筆移除該子專案在本軍師下的所有角色代碼登記，同步更新軍師 CLAUDE.md 關聯專案表與全域反向註冊表；移除前掃描未完成交接文件，警告即將失效的角色代碼。

---

## 問題框架

`kunsu-init` 的 `add-project` 子指令支援新增子專案登記與角色代碼異動，但沒有對應的移除路徑。子專案可能因檔案結構合併（多個子專案併入同一個 repo）或拆分（一個子專案拆成多個新 repo）而使原本登記的路徑或角色代碼失效；目前唯一手段是手動編輯 `~/.claude/kunsu-registry.json` 與軍師 CLAUDE.md，既不安全（JSON 手改容易破壞格式）也無法比照既有流程對未完成交接文件示警。`kunsu-list` 已具備 stale 偵測能力（⚠ 標記路徑不存在的登記）但只報不修，缺一個對應的「修」動作收尾。

---

## 關鍵決策

- **只做軍師端發起，不做子專案端「退出申請」投遞** — 移除通常是軍師先發現舊登記失效（子專案路徑已合併／拆分／消失），子專案端此時未必還能正常執行任何 skill；比照 `kunsu-apply` 建反向投遞流程無實益。
- **整筆移除，不支援單一角色代碼選擇性保留** — 子專案在同一軍師底下多半只掛單一角色代碼，為少見的多角色部分退場場景增加選取介面複雜度不划算。
- **清單呈現採失效感知輔助選取** — 比照 `kunsu-list` 既有 stale 偵測邏輯（路徑存在性查核），路徑已不存在的登記標記並排列在前；直接命中「合併/拆分後路徑失效、不確定該刪哪筆」的實際情境，複用既有邏輯無新技術風險。
- **移除前掃描未完成交接文件** — 比照 `add-project` 步驟⑨角色改名警告的做法：角色代碼一旦從註冊表移除，`to:` 等於該代碼且尚未完成的交接文件會變成永遠不會被 `/kunsu-inbox` 掃到的孤兒；只提示不阻擋，維持「軍師定案」的既有精神。
- **移除需要明確的不可逆確認，力度高於一般確認 commit** — 全域註冊表本身不受任何 repo 版控，一旦寫入無法像 CLAUDE.md 一樣靠 git 復原。

---

## 關鍵流程

- F1. 移除子專案登記
  - **觸發：** 使用者在軍師 repo 執行 `remove-project`（身分驗證比照 `add-project` 步驟①，需為軍師根目錄）。
  - **步驟：** 讀取本軍師在全域註冊表與 CLAUDE.md 關聯專案表的登記清單 → 對每筆執行路徑存在性查核（stale 偵測）→ 呈現清單（stale 標記並排前）供選取 → 選定後掃描 `docs/handoffs/` 頂層，篩選 `to:` 等於待移除角色代碼且未完成（無回覆，或最新回覆 status 非 `done`）的交接文件 → 有則列清單提示，使用者確認是否繼續 → 明確不可逆確認 → 移除 CLAUDE.md 關聯專案表對應列與環境限制小節（若有）、移除全域註冊表對應子專案在本軍師的 entry（僅登記本軍師時整個 key 移除；仍登記其他軍師時只移除本軍師對應的陣列元素）→ 回報移除結果 → 依 ADR 009 確認 commit 慣例，詢問是否 commit CLAUDE.md 異動（registry 不進版控，不受此步驟影響）。
  - **結果：** 該子專案在本軍師的登記完全消失（CLAUDE.md 與 registry 同步），既存 `docs/handoffs` 與其回覆歷史檔案不受影響。
  - **涵蓋 R1–R9。**

---

## 需求

- R1. `remove-project` 僅能在軍師根目錄的 session 執行，身分驗證比照 `add-project` 步驟①（比對全域註冊表的 kunsu 路徑）。
- R2. 呈現本軍師目前登記的子專案清單供選取，對每筆執行路徑存在性查核，路徑已不存在者標記並排列在清單前段。
- R3. 選定移除時，同軍師下該子專案登記的所有角色代碼一併移除，不支援單一角色代碼選擇性保留。
- R4. 執行實際刪除前，掃描 `docs/handoffs/` 頂層文件，篩選 `to:` 等於待移除角色代碼且未完成（無回覆，或最新回覆 status 非 `done`）的交接文件，列出清單提示使用者。
- R5. 未完成交接警告只提示、不自動阻擋；使用者可選擇繼續移除或取消。
- R6. 移除前需經一次明確的不可逆確認，措辭須說明「全域註冊表不受版控、無法用 git 復原」，確認後才寫入。
- R7. 移除同步更新兩處：軍師 CLAUDE.md 關聯專案表對應列（含環境限制小節，若有）與全域反向註冊表對應子專案在本軍師的 entry。
- R8. 移除不刪除任何既存的 `docs/handoffs` 或 `docs/handoffs/replies` 檔案，僅停用未來 `/kunsu-inbox` 對該角色代碼的路由。
- R9. 移除完成後，依 ADR 009 既有確認 commit 慣例處理 CLAUDE.md 異動；registry 不進任何 repo 版控範圍，不受此步驟影響。

---

## 驗收例

- AE1. 子專案僅登記本軍師，移除後整個 key 消失
  - **涵蓋 R7。**
  - **Given** 該子專案的 registry entry 只有一筆（本軍師），**When** 使用者確認移除，**Then** registry 中該子專案的 key 整個被移除，不留空陣列。
- AE2. 子專案同時登記多個軍師，只移除本軍師
  - **涵蓋 R7。**
  - **Given** 該子專案的 registry entry 陣列含本軍師與另一軍師兩筆，**When** 使用者在本軍師確認移除，**Then** 只移除本軍師對應的陣列元素，另一軍師的登記保留不動。
- AE3. 有未完成交接文件時的警告
  - **涵蓋 R4、R5。**
  - **Given** 待移除角色代碼有一份未完成（無回覆）的交接文件，**When** 使用者執行移除，**Then** 先列出警告清單，使用者確認後才繼續寫入，不自動阻擋。
- AE4. 無登記可移除
  - **涵蓋 R2。**
  - **Given** 本軍師目前沒有任何已登記子專案，**When** 使用者執行 `remove-project`，**Then** 提示無可移除項目，不進入選取步驟。

---

## 範圍邊界

- 子專案端主動投遞「退出申請」（對稱 `kunsu-apply` 反向流程）——不在本次範圍，軍師端手動移除已足夠涵蓋合併/拆分情境。
- 單一角色代碼選擇性保留（部分退場）——不支援；有此需求時，使用者可先整筆移除、再以 `add-project` 重新登記其餘角色。
- registry 移除前的稽核快照或復原機制——維持現行「registry 無版控、無稽核歷史」慣例，不在本次新增。
- 合併/拆分情境的自動偵測或新舊登記自動合併（例如自動判斷兩個路徑其實是同一專案）——不在本次範圍；`remove-project` 只負責移除舊登記，新增登記仍走既有 `add-project`。

---

## 依賴與假設

- 依賴 `kunsu-list` 既有的 stale 偵測邏輯（路徑存在性查核）可直接複用於清單呈現。
- 依賴 `add-project` 步驟⑨既有的未完成交接文件掃描邏輯（比對 `to:` 與最新回覆 status）可直接複用。
- 假設單機單使用者情境，移除操作無並發寫入疑慮，與 `registry-merge.sh` 現行假設一致。

## 開放問題

**留待規劃階段決定：**

- 子指令確切命名與觸發語（`remove-project` 或其他中文口語，如「移除子專案」「刪除子專案登記」）。
- 不可逆確認的具體措辭與呈現形式（`AskUserQuestion` 兩階段確認，或單次強措辭確認）。
- 是否新增獨立 ADR（如 ADR 012）記錄此決策，或併入既有相關 ADR 系列文件。
- registry 寫入的實作方式：新增 `registry-remove.sh` 腳本（對稱 `registry-merge.sh`）或在既有腳本擴充 `--remove` 旗標。
