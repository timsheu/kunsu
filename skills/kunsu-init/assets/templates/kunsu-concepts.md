# Concepts

> 共享領域詞彙表——實體、具名流程與具備專案特定意義的狀態概念。以核心領域詞彙起種，之後隨 ce-compound／ce-compound-refresh 處理學習而持續累積；也歡迎直接編輯。純詞彙表，不是規格書或包山包海的雜記。

## 跨專案協調

### 軍師（Kunsu / Planner）
本專案自身的角色：純文件、不含程式碼的規劃協調中心，負責在多個獨立子專案之間規劃跨專案功能、產出交接文件、彙整回覆。只唯讀查閱子專案的程式碼與文件以評估可行性，絕不寫入子專案的檔案系統——這個唯讀邊界是軍師存在的核心不變量，不是慣例。

### 子專案（Sub-project）
軍師協調的對象，各自擁有獨立 git repo 與自己的專案規範文件。軍師對子專案只有唯讀存取權，不做任何寫入。

### 交接文件（Handoff）
軍師（或任何發起方）產出的獨立文件，記錄需要另一個 session 或角色接手研究／實作的議題。建立後即為定案快照：任何人（含發起方自己）都不再編輯這份文件；後續狀態更新只透過對應的回覆信箱表達，不回頭修改文件本體。生命週期以狀態表示：待接手、接手方研究中、已完成，完成後歸檔保存。

### 回覆信箱（Reply Inbox）
交接文件的附屬回覆機制：接手方完成後新增一個獨立檔案回報結論，而不是編輯交接文件本體。同一份交接文件可分階段回覆多次，每次都是新檔案，append-only、不覆蓋前次回覆。核心規則是「任何檔案永遠只有一個作者」，用來防止多方共編同一份可變文件造成版本漂移。軍師只讀取回覆信箱，不編輯回覆檔案本身。

### 定案規劃（Authoritative Plan）
軍師彙整各子專案文件、直接查證程式碼後產出的跨專案功能規劃，明確取代各子專案自己的原始規劃文件成為唯一權威版本。子專案間對規格的矛盾或落差在這裡訂正定案，不是並列兩邊各自的說法；後續發現的缺口或決策，也回頭補進這份文件，不留在只有單一 session 知道的地方。
