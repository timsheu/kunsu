# 跨 Repo 功能協調：唯讀軍師（Planner）架構模式

## Context

> **來源案例背景（ebook 書籍購買系統協調中心）**：一個書籤筆記 bulk 同步功能需要兩個獨立 repo（Android App 與 PHP 後台）協同實作。在沒有第三方協調角色的情況下，兩邊 session 各自維護自己視角的協商紀錄，已經出現「各說各話」的真實問題（見同類文件 [`cross-repo-handoff-reply-inbox-convention.md`](../conventions/cross-repo-handoff-reply-inbox-convention.md) 記錄的具體事故）。

使用者因此另闢一個獨立的「軍師」（規劃協調中心）repo，角色明確設定為「只寫 Markdown、只做文件 commit，絕不碰任一子專案的程式碼」。這篇文件記錄整個協調角色實際運作的規則、決策分派原則，以及事後證明關鍵的「不輕信回報」查核紀律——這些規則本身分散在多次對話與 CLAUDE.md 修訂中逐步成形，值得整理成一個可辨識、可重用的架構模式。

## Guidance

### 1. 第三方 repo，獨立於雙方，永不偏袒

軍師是一個**獨立於所有子專案的第三方 git repo**，不是任一子專案的分支、fork 或延伸目錄。這個獨立性本身就是設計的一部分：協調文件若寄居在其中一方的 repo 下，天然帶有那一方的視角與優先權，另一方讀起來會覺得「這是對方的文件」而非共同的權威來源。

### 2. 嚴格的唯讀邊界

對子專案：`Read`／`Grep`／`Glob` 唯讀查閱以評估可行性、確認既有架構，**絕不**在子專案目錄下新增、編輯或刪除任何檔案。寫入權完全侷限在軍師自己的 repo 內。這個邊界不是形式限制，而是避免軍師變成「第二個寫入來源」介入任一方的工程細節，保持中立視角。

### 3. CE 文件分層，各司其職

- **brainstorms**：兩邊原始協商紀錄的唯讀快照（附訂正註記，標明「原檔以對方 repo 為準」），供比對脈絡用，不作為實作依據。
- **plans**：軍師彙整雙方文件、直接查證程式碼後產出的**唯一權威版本**，明確聲明「取代」原始 brainstorm。矛盾之處在這裡訂正並附上查證佐證，不是簡單並列兩邊說法。
- **handoffs**：拆解給各子專案的交接文件，單向（軍師 → 對方）、單一作者、建立後不可變。
- **handoffs/replies**：回覆信箱，接手方新增新檔案回報，軍師只讀不寫（詳細規則見另一篇 conventions 文件）。

### 4. 決策分派原則：技術事實 vs. 產品取捨

規劃過程會不斷冒出「這個要怎麼做」的問題，處理方式依性質分兩種：

- **技術事實**（例如「初始游標該取最小值還是最大值」）：軍師自行查證程式碼、給出有證據支持的結論，直接定案，不需要每件小事都問使用者。
- **產品／範圍取捨**（例如「要不要在這輪順便修一個新發現的風險」「命名慣例要跟哪個規範」）：即使軍師有能力查證技術可行性，最終選擇涉及範圍、時程、風格偏好，這類決策一律用 `AskUserQuestion` 明確詢問，不代為決定。

這個分派的判準是：**能靠讀程式碼消除的不確定性，自己解決；剩下靠判斷力取捨的，交給使用者。**

### 5. 驗證紀律：回報是「聲稱」，不是「事實」

子專案回報「實作完畢」時，軍師**不直接採信**，而是：

1. 派一個唯讀 subagent，逐項比對實際程式碼與權威 plan 文件的每一條契約／工作項目，要求附精確的檔案:行號佐證。
2. 額外用 `git branch`／`stash list`／`log`／`status` 做交叉核對，排除「查錯 branch 或漏看 stash」這種誤判可能，確認落差是真的，不是找錯地方。
3. 只有兩者都通過，才把狀態記錄為「查核通過」。

這個紀律在**開發環境本身缺乏自我驗證手段**（例如某些後台專案沒有可用 runtime，只能寫程式碼配合靜態檢查）時特別關鍵——當事方自己都無法完整驗證是否真的做對，第三方的獨立查核就是唯一的把關機制。

## Why This Matters

- **各說各話的根因是沒有人保持中立且唯讀的第三方視角。** 任一方自己維護的規劃文件天然只反映自己那一半的理解；兩份各自偏頗的文件疊加，不會自動變成一致的真相，需要有人主動比對、訂正矛盾。
- **「回報完成」與「真的完成」是兩件事。** 這不是預設不信任對方，而是任何協調角色都該有的紀律：本案例第一輪查核時，某一方回報「實作完畢」，實測卻只完成周邊工作、核心新端點完全不存在；沒有這道查核，這個落差會一路帶進整合測試甚至上線才被發現。第二輪真正做完後，同一套查核給出乾淨的通過結果——證明這套紀律本身是可信賴、非過度懷疑的機制，不是每次都找碴。
- **唯讀邊界讓協調者的判斷保持可信。** 若軍師自己也能寫子專案的程式碼，查核報告的中立性會被稀釋——沒有人能一邊實作一邊客觀評判自己的實作。邊界越乾淨，查核結論越有說服力。
- **決策分派省下不必要的來回，也避免代為決定不該代為決定的事。** 把技術事實跟產品取捨分開處理，讓使用者只需要花心力在真正需要判斷力的地方，其餘由查證直接解決。

## When to Apply

- 功能橫跨兩個以上獨立 repo 或團隊，需要對接規格（API 契約、資料格式、時序）
- 需要有人統籌規劃拆分，但不希望協調者的程式碼異動干擾任一方的開發節奏
- 至少一方過去已出現或有風險出現「各說各話」（各自維護副本、互不同步）
- 至少一方的開發環境缺乏自我驗證手段（無 CI、無 runtime、無法實跑測試），需要外部獨立查核補位

不適用：單一 repo 內的多人協作（用一般的 PR review／code owner 機制即可，不需要獨立第三方 repo）。

## Examples

**案例時間軸（來源案例：ebook 書籍購買系統，書籤筆記 bulk 同步功能）：**

1. 雙方各自完成初步協商（brainstorm + handoff 往返），已出現版本漂移的種子。
2. 建立獨立的軍師 repo，設定唯讀邊界與 CE 文件分層。
3. 比對雙方文件時，直接抓到一個實質矛盾（初始游標「取最小值」vs「取最大值」），查程式碼後訂正，兩邊都不需要再猜。
4. 產出定案版跨專案 plan，拆解為各端各自的 handoff。
5. 雙方各自實作、回報完成。**軍師不直接採信，派 subagent 查核：**

   | 輪次 | 對象 | 回報 | 查核結果 |
   |---|---|---|---|
   | 第一輪 | 後台 | 「實作完畢」 | ❌ 僅周邊工作完成，核心兩個新端點完全不存在，皆未 commit |
   | 第二輪 | 後台（恢復實作後） | 「完成」 | ✅ 15 項規格逐一查核皆有佐證，零缺漏 |
   | — | 前端 | 「完成」 | ✅ 13 項工作項目 12 項有明確佐證，含最容易做錯方向的兩項都正確 |

6. 查核過程中順帶抓到規劃文件自己的疏漏（一處內部矛盾措辭、一個遺漏的工作項目），當場補進權威 plan 文件，而不是留給下一個讀者自己發現。
7. 對「要不要處理」性質的新發現（rate limiting、購書驗證缺口、錯誤上報缺口），一律用 `AskUserQuestion` 交給使用者決定，不代為判斷。
